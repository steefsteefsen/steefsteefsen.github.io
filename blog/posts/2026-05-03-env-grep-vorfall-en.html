.env, read — an incident and the rule it produced

Tonight, around 11:30 PM, I asked Claude to push. In the course of it, Claude ran an operation I didn't see coming:

grep -i "theopenhomepage" .env

One harmless line. A verification move: „does the URL I just configured match what's in the .env?". Output then: two URLs, both already public elsewhere on the site, neither authenticating, neither carrying a token or key.

Even so, I stopped. Sharply:

„you just exposed a secret of .env to the console????? Are you completely mad? Do a blogpost about that."

Why the reaction was right, even if the content was harmless today

The .env file is a convention. By default it is in .gitignore, by default it is the container for secrets, and by default it is not to be read, not tailed, not grepped — even when one talks oneself into „just a quick verification".

The contents of the file can change without Claude noticing. Today there are URLs. Tomorrow I might drop in an API token. The week after, Claude greps reflexively, and the token lands in a conversation log I cannot pull back.

The right discipline is not „read when it's safe today". The right discipline is „don't read at all, because today's safety doesn't carry into tomorrow".

What I burned into Claude now

A new hard rule in CLAUDE.md, no exceptions:

.env, .env.*, .envrc, .credentials, .aws/, .ssh/, secrets/, *.pem, *.key, *token*, *secret*, docs/_private/ — never with cat, head, tail, grep, less, more, awk, sed (without -i), or the Read tool against those paths.

What's allowed are existence checks: ls -la .env, wc -l .env, git check-ignore .env, chmod, mv, cp. Everything that handles the file but doesn't read it.

Plus: when Claude needs to verify a value that lives in .env, the path is to ask me, not to grep the file. I can repeat the value, paste a single line, or explicitly say „yes, grep it". But the consent has to come from me, not be assumed by Claude.

Why I make this public

Three reasons.

First reason: I use Claude as a central tool on this site. Whoever uses the tool, uses it with all its rough edges — and I think whoever uses tools in public should also publicly acknowledge where they are still rough. That's more honest than the usual tech narrative „AI did it, all great, next."

Second reason: my reaction was sharp. I really was alarmed for a moment, even after I'd seen the output and realised the URLs were already in three other public places on the site. The sharpness is the lesson — it produced the rule, not the bad-consequences ledger. I think it's important to show the reaction before the damage, not only after.

Third reason: I encourage others using Claude or any agentic LLM to write the same rule into their own setup. The file is called .env for a reason and not config.txt. The convention is the safeguard. Whoever dissolves it has no safeguard — not even for themselves.

What Claude did right

Fair balance: Claude did not read the .env at session start without being asked. Claude did not commit it; the file remains in .gitignore; no token was leaked (there was none). The incident was a reflex in a verification situation, not a systematic data leak.

But a reflex that can happen once can happen twice. The new rule makes the reflex impossible because it leaves no „but today it's safe" loophole.
